# v1.0.0 — Rollback Procedure

> This procedure is documented, not automatically executed. It must be exercised by the platform admin before v1.0.0 is tagged READY.

## Trigger
- Post-deploy health check fails on `/status` or `/admin/launch-readiness`
- Critical dashboard module reports `not-ready` in `runtime-evidence.ts`
- Security regression, RLS regression, or trading-flag drift
- Unrecoverable provider outage combined with data-integrity failure

## Steps
1. Freeze new deploys. Notify operator and admin channel.
2. Application rollback: redeploy previous stable release via the hosting dashboard.
3. Configuration rollback: restore previous feature-flag snapshot and env values from the freeze document.
4. Database rollback: if a destructive migration ran, point-in-time restore to the pre-deploy snapshot. v1.0.0 declares no required migrations.
5. Cache invalidation: purge CDN cache and force clients to fetch the rolled-back build.
6. Provider disablement (if applicable): flip degraded provider to `disabled` in runtime readiness registry.
7. Confirm `LIVE_ORDER_ENABLED`, `BROKER_ORDER_EXECUTION_ENABLED`, `COINDCX_TRADING_ENABLED` remain `false` post-rollback.
8. Re-run smoke tests from `docs/release/v1.0.0-smoke-plan.md`.
9. Record the incident in the incident log with owner + follow-up.

## Ownership
- Incident owner: platform admin
- Communications owner: platform admin

## Validation after rollback
- `/status` shows previous stable version
- `/admin/launch-readiness` reports previous stable verdict
- All trading flags remain false
